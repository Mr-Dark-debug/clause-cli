Getting Started

Create your first AI-governed project in under a minute.

What is Clause?

Clause is an AI-native project scaffolding tool. It generates a project structure that includes a special ai_prompt_guidelines/ directory ‚Äî a set of governance files that teach AI coding assistants how to behave in your codebase.

Think of it as a constitution for your AI: coding standards, architecture decisions, and reflection checkpoints ‚Äî all in one place.

Who should use Clause?

  • Developers using AI coding assistants (Copilot, Cursor, Claude, etc.)
  • Teams that want consistent, governed AI output
  • Open-source maintainers who want AI-friendly repos

Your First Project

Step 1 ‚Äî Install Clause

If you haven't already, install Clause using your preferred method:

bash
# macOS / Linux
curl -fsSL https://clause.dev/install.sh | bash

# Windows
winget install Clause.ClauseCLI

Step 2 ‚Äî Initialize a project

Run the interactive wizard:

bash
clause init my-awesome-project

You'll be guided through an interactive TUI that asks about:

  • Project type (web app, API, CLI, library, etc.)
  • Tech stack (React, Go, Python, etc.)
  • AI governance preferences
  • Git initialization and README generation

Step 3 ‚Äî Explore the output

bash
cd my-awesome-project
tree -L 2

Your project now contains an ai_prompt_guidelines/ directory that any compatible AI assistant can read to understand your project's rules, architecture, and conventions.

Understanding the Output

Key Files

  • ai_prompt_guidelines/system_prompt.md ‚Äî The main governance file that tells AI how to behave in your project
  • ai_prompt_guidelines/architecture.md ‚Äî Documents your project's architecture decisions
  • ai_prompt_guidelines/brainstorm.md ‚Äî A self-reflection scratchpad the AI uses before writing code
  • ai_prompt_guidelines/component_registry.json ‚Äî Registry of all known components and their relationships

How AI Uses These Files

When you open your project in an AI-assisted editor, the assistant reads the governance files and:

  1. Understands your coding standards and conventions
  2. Follows the architecture patterns you've defined
  3. Uses the brainstorm system to reason before generating code
  4. Checks the component registry to avoid duplicating work
